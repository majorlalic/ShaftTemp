## 1.sql适配达梦
- 建表语句之前不兼容达梦，目前表建了，但是表注释和字段注释和索引需要单独的sql补充
- 持久层中的sql也许考虑达梦数据库的语法问题

## 2. 介绍目前分controller的逻辑
- controller分割的逻辑是什么，alarmdoccontroller什么意思，我觉得应该分以下
    - iot服务接收
    - area，org
    - 监控对象管理
    - 设备管理
- 为什么现在都是用的构造函数，而不是用@service等注入组件

## 3. controller返回包装复杂
- 现在是ResponseEntity.ok(ApiResponse.success(alarmViewService.statistics(areaId, deviceId, startTime, endTime))); 层数太多
- 我要移植的项目用的是RestObject.newOk(); 可以去掉ResponseEntity.ok，将ApiResponse.success改为RestObject.newOk()

## 4. entity命名改为*VO
- 比如说AlarmEntity改为AlarmVO
