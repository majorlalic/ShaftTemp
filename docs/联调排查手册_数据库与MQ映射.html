3. 常见报错排查与解决

3.1 报错：partition binding not found for partitionCode=xxx

原因：设备/分区映射未命中，常见于 iotCode 或 dataReference 不一致。

排查 SQL：

-- 1) 设备是否存在
SELECT id, iot_code FROM ODS_DWEQ_DM_DEVICE_D
WHERE iot_code IN ('shaft-dev-01', 'shaft-dev-001') AND deleted = 0;

-- 2) 指定设备+分区是否存在绑定
SELECT d.iot_code, b.partition_id, b.partition_code, b.data_reference
FROM ODS_DWEQ_DM_MONITOR_PARTITION_BIND_D b
JOIN ODS_DWEQ_DM_DEVICE_D d ON d.id = b.device_id
WHERE d.iot_code = 'shaft-dev-001'
  AND b.partition_id = 10
  AND b.deleted = 0;

-- 3) 按 dataReference 精确核对
SELECT b.*
FROM ODS_DWEQ_DM_MONITOR_PARTITION_BIND_D b
WHERE b.data_reference = '/TMP/shaft-dev-001_TMP_th10'
  AND b.deleted = 0;

解决：

1) 统一 iotCode 格式（推荐 shaft-dev-001，非 shaft-dev-01）。

2) 统一 dataReference 大小写和格式（TMP/th 位数一致）。

3) 补齐 monitor_partition_bind 绑定数据。

3.2 报错：there was no typehandler found for parameter collecttime

原因：MyBatis 参数名/类型不一致，常见是写成 collecttime 而非 collectTime。

排查：检查 mapper XML 占位符与 VO 字段名是否一致。

解决：使用 #{collectTime,jdbcType=TIMESTAMP}，批量插入写 #{item.collectTime,...}。

3.3 报错：唯一约束冲突（uk_alarm_merge_key）

原因：并发创建同一 merge_key 告警。

排查：

SELECT id, merge_key, status, merge_count, updated_on
FROM ODS_DWEQ_DM_ALARM_D
WHERE merge_key IS NOT NULL
ORDER BY updated_on DESC;

解决：保持 createOrMerge 冲突兜底逻辑开启；仅待确认主单保留 merge_key。

3.4 现象：压测显示成功但数据库无数据

原因：仅看 HTTP 200，未校验业务 success/code；或压到了不落 raw_data 的接口。

排查：

1) 确认压测接口是 /shaft/iot/reports/measure（会落 raw_data）。

2) 检查响应体 code/success，不只看 http status。

3) 检查日志是否有业务异常。

3.5 报错：不能修改聚集索引列（修改 alarm.id 类型）

原因：达梦限制，主键列（聚集索引）不能直接 MODIFY 类型。

解决：采用重建表迁移法：

1) 新建新结构表（id 为 varchar）。

2) INSERT SELECT 迁移（Long 字段 TO_CHAR）。

3) 重建索引与约束。

4) 旧表备份重命名，新表改正式名。

4. 联调前最小检查清单

1) 基础表有数据：device、monitor、monitor_partition_bind、alarm_rule。

2) iotCode 与分区映射一致：设备编码位数和大小写一致。

3) 告警列表筛选参数可用：areaName、status、alarmTypeBig、startTime/endTime。

4) 规则启用状态正确（enabled=1）。

5) 先用单条 measure 请求验证，再跑压测脚本。
