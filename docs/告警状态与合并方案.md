# 告警状态与合并方案

本文档用于明确后续告警模型的业务规则，重点包括：

- 告警状态定义
- 告警合并规则
- 事件记录规则
- 状态码是否使用 `0/1/2/3`
- 接口返回要求

该方案用于指导后续代码改造。

## 1. 设计目标

告警模型需要满足以下业务要求：

1. 同一个监测对象，即使不同楼层触发，也可以合并到同一条告警
2. 只有“待确认”状态的告警允许继续合并
3. 已确认、持续观察、误报状态的告警不再合并
4. 后续同一监测对象再次触发时，应产生新的告警主单
5. 可查看一条告警合并过程中的所有事件
6. 告警列表可查询：
   - 首次告警时间
   - 最新告警时间
   - 告警内容
7. 事件也需要记录告警内容

## 2. 总体模型

继续采用：

- `alarm`
  告警主单
- `event`
  告警过程事件流水

语义如下：

### 2.1 `alarm`

表示某个监测对象下某一类告警的一次告警单。

一条主单聚合的是“同一监测对象 + 同一告警类型 + 可合并状态窗口内”的告警过程。

### 2.2 `event`

表示主单过程中的一次动作或一次触发。

包括：

- 首次触发
- 合并触发
- 人工确认
- 改为持续观察
- 判定误报
- 后续恢复
- 关闭

## 3. 告警状态定义

建议使用“数据库存数字码，代码中用枚举常量”的方式。

## 3.1 推荐状态码

- `0 = PENDING_CONFIRM`
  待确认

- `1 = CONFIRMED`
  已确认

- `2 = OBSERVING`
  持续观察

- `3 = FALSE_POSITIVE`
  误报

后续如需扩展，可以继续往后增加：

- `4 = CLOSED`
  已关闭

- `5 = AUTO_RECOVERED`
  自动恢复

## 3.2 为什么推荐存数字

推荐数据库中使用 `tinyint` 或 `int` 存储状态码，原因如下：

- 存储更紧凑
- 查询和索引更稳定
- 文案变更不会影响数据库值
- 接口层可以同时返回：
  - `statusCode`
  - `statusName`

## 3.3 为什么不能只用裸数字

虽然数据库适合存数字，但代码里不能到处直接写 `0/1/2/3`。

正确方式是：

- DB：存数字
- Java：用枚举或常量映射
- API：返回数字码和中文名

例如：

- `statusCode = 0`
- `statusName = "待确认"`

## 4. 告警合并规则

## 4.1 合并维度

后续建议合并维度改成：

- `monitor_id`
- `alarm_type`
- `status = PENDING_CONFIRM`

不再把：

- `shaft_floor_id`
- `partition_code`

作为主单拆分条件。

它们保留在事件里，用于描述“这次是谁触发的”。

## 4.2 具体规则

### 规则 1

同一个监测对象，即使不同楼层触发，也合并。

例如：

- `1号竖井-3楼` 触发定温告警
- `1号竖井-5楼` 再次触发定温告警

如果当前已有一条“1号竖井”的同类告警主单且状态为待确认，则两次触发应并入同一条告警主单。

### 规则 2

只有待确认状态允许合并。

也就是说，只有 `status = 0(PENDING_CONFIRM)` 的主单才会继续吸收新的同类事件。

### 规则 3

已确认、持续观察、误报状态不再合并。

如果旧主单状态为：

- `1 = CONFIRMED`
- `2 = OBSERVING`
- `3 = FALSE_POSITIVE`

则后续同一监测对象再出现同类告警时，必须新建新的 `alarm` 主单。

### 规则 4

所有事件都要可追溯。

无论是否合并，每一次触发都应保留在 `event` 中。

因此：

- 主单负责承载“当前告警汇总状态”
- 事件负责承载“完整过程”

## 5. 告警流程

## 5.1 首次触发

当某个监测对象首次出现某类告警时：

1. 新建 `alarm`
2. `status = 0(PENDING_CONFIRM)`
3. `first_alarm_time = 当前时间`
4. `last_alarm_time = 当前时间`
5. `merge_count = 1`
6. `content = 本次告警内容`
7. 写入一条 `event`

## 5.2 待确认状态下再次触发

若已存在同一监测对象、同一告警类型、状态为待确认的主单：

1. 不新建 `alarm`
2. 更新原主单：
   - `last_alarm_time`
   - `merge_count + 1`
   - `content` 更新为最近一次触发内容
3. 写一条新的 `event`

## 5.3 人工确认

处理规则：

1. `alarm.status = 1(CONFIRMED)`
2. 写一条确认事件

## 5.4 持续观察

处理规则：

1. `alarm.status = 2(OBSERVING)`
2. 写一条状态变更事件

## 5.5 误报

处理规则：

1. `alarm.status = 3(FALSE_POSITIVE)`
2. 写一条状态变更事件

## 5.6 后续再次触发

如果旧主单已不处于待确认状态，则：

1. 不合并旧单
2. 新建一条新的 `alarm`
3. 新单重新从待确认状态开始

## 6. 告警内容规则

## 6.1 `alarm.content`

建议定义为：

- 主单最近一次触发的摘要内容

用于：

- 告警列表展示
- 告警详情主信息展示

示例：

`3楼温度为75度，阈值为50度`

## 6.2 `event.content`

建议定义为：

- 该条事件对应的具体内容

用于：

- 事件时间线展示
- 查看每次合并触发时的具体楼层、温度和阈值

示例：

- 第一次事件：
  `3楼温度为75度，阈值为50度`
- 第二次合并事件：
  `5楼温度为82度，阈值为50度`

## 6.3 结论

后续应同时保留：

- `alarm.content`
- `event.content`

这样既能在列表里看“当前摘要”，也能在详情里看“完整过程”。

## 7. 事件模型建议

## 7.1 事件类型

建议为 `event` 增加单独事件类型字段。

可选两种存法：

- 字符串
- 数字码

如果统一用数字码，推荐如下：

- `0 = TRIGGER`
  首次触发

- `1 = MERGE`
  合并触发

- `2 = CONFIRM`
  人工确认

- `3 = OBSERVE`
  改为持续观察

- `4 = FALSE_POSITIVE`
  判定误报

- `5 = RECOVER`
  恢复

- `6 = CLOSE`
  关闭

## 7.2 事件应记录哪些内容

每条 `event` 至少应记录：

- `alarm_id`
- `event_type`
- `event_time`
- `content`
- `monitor_id`
- `shaft_floor_id`
- `partition_code`
- `trigger_value`
- `threshold`
- `merged_flag`
- `detail_json`

其中：

- `content` 用于直接展示
- `detail_json` 用于扩展字段

## 8. 数据库存储建议

## 8.1 `alarm.status`

建议：

- 类型改为 `tinyint`

替代当前字符串状态。

## 8.2 `event`

建议补充字段：

- `event_type tinyint`
- `content varchar(1000)`

如果不立刻改表，临时也应保证：

- `event.detail_json` 中有 `content`

但长期建议还是单独字段化。

## 9. 接口返回建议

## 9.1 告警列表

建议至少返回：

- `alarmId`
- `alarmType`
- `statusCode`
- `statusName`
- `firstAlarmTime`
- `lastAlarmTime`
- `mergeCount`
- `content`

## 9.2 告警详情

建议返回：

- 主单信息
- 事件列表

## 9.3 事件列表

建议每条返回：

- `eventId`
- `eventType`
- `eventTypeName`
- `eventTime`
- `content`
- `floorName`
- `partitionCode`
- `triggerValue`
- `threshold`
- `mergedFlag`

## 10. 对当前实现的影响

如果按本方案改造，核心变化如下：

### 10.1 告警主单合并条件变化

从当前的：

- `monitor_id + partition_code + alarm_type + ACTIVE/CONFIRMED`

改成：

- `monitor_id + alarm_type + status = PENDING_CONFIRM`

### 10.2 状态体系变化

从当前偏技术状态：

- `ACTIVE`
- `CONFIRMED`
- `RECOVERED`
- `CLOSED`

改成偏业务状态：

- `0 = PENDING_CONFIRM`
- `1 = CONFIRMED`
- `2 = OBSERVING`
- `3 = FALSE_POSITIVE`

### 10.3 事件模型增强

需要保证事件可独立展示内容，因此应增加：

- `event_type`
- `content`

### 10.4 查询接口增强

告警查询应明确返回：

- 首次告警时间
- 最新告警时间
- 告警内容
- 合并次数
- 完整事件列表

## 11. 推荐结论

建议最终采用以下方案：

1. `alarm.status` 使用数字码存储
2. 代码层统一用枚举映射状态码
3. 合并规则只认：
   - 同一监测对象
   - 同一告警类型
   - 主单状态为待确认
4. 楼层和分区放到 `event` 里描述每次触发来源
5. `alarm.content` 保存主单最近一次告警内容
6. `event.content` 保存每次事件内容
7. 所有合并过程都可通过事件列表完整追溯
