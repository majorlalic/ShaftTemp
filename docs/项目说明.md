# 测温物联网系统项目说明

## 1. 项目现状

当前仓库已经完成一版可联调、可演示的最小可用实现，面向“分区 MQ 测温消息”模型，而不是早期的一维数组模型。

当前技术实现口径：

- 持久层统一使用 MyBatis Mapper
- 实时消息接入协议为 MQTT
- MQTT 默认关闭，本地启动不依赖 broker
- 工程已拆分为 `service` + `iot` 双模块

系统重点能力包括：

- 分区测温消息接收
- Redis 实时状态缓存
- `ODS_DWEQ_DM_RAW_DATA_D` 原始快照落库
- 规则告警生成
- 告警主单合并
- 告警事件流水
- 离线巡检任务
- 查询接口
- 告警规则 CRUD
- 终端视图接口
- 组织树查询、模板下载、CSV 导入
- 全局规则动态生效

当前 `service` 模块的 Controller 已按 5 个业务域收口：

- IoT 服务接收：`IotIngestController`
- area/org：`AreaOrgController`
- 监控对象管理：`MonitorController`
- 设备管理：`DeviceController`
- 告警管理：`AlarmController`

## 2. 当前业务模型

### 2.1 消息模型

系统按两类消息处理：

- `Measure`
  分区测量摘要，包含 `MaxTemp`、`MinTemp`、`AvgTemp` 等
- `Alarm`
  分区状态摘要，包含 `AlarmStatus`、`FaultStatus`

当前 MQTT 订阅主题默认配置为：

- `/TMP/+/Measure`
- `/TMP/+/Alarm`

### 2.2 绑定模型

一个设备下有多个分区，每个分区通过 `ODS_DWEQ_DM_MONITOR_PARTITION_BIND_D` 绑定到：

- 监测对象 `monitor`
- 楼层 `ODS_DWEQ_DM_SHAFT_FLOOR_D`
- 分区编码 `partition_code`
- MQ 引用 `data_reference`

### 2.3 告警模型

告警采用：

- `alarm`
  告警主单
- `event`
  告警过程事件

主单和事件的语义已经拆开：

- `merge_count`
  累计触发次数
- `event_count`
  已写入的关键事件条数

## 3. 告警状态

当前告警状态码如下：

- `0`
  待确认
- `1`
  已确认
- `2`
  持续观察
- `3`
  误报
- `4`
  已关闭
- `5`
  自动恢复

系统接口返回时会同时给出 `statusCode` 和 `statusName`。

## 4. 告警合并规则

当前规则已经按业务要求实现：

- 同一监测对象、同一告警类型，在 `待确认` 状态下合并
- 不按楼层拆主单，不按分区拆主单
- `已确认 / 持续观察 / 误报 / 已关闭 / 自动恢复` 状态不再继续合并
- 后续同监测对象同类告警会新建新的待确认主单

为保证并发下不重复建待确认主单，系统使用：

- `alarm.merge_key`
- `UNIQUE KEY uk_alarm_merge_key (merge_key)`

其中：

- 待确认主单：`merge_key = monitorId:alarmType`
- 非待确认主单：`merge_key = NULL`

这保证了数据库层只允许存在一条可合并的待确认主单。

## 5. 实时数据处理链路

### 5.1 Measure 链路

`Measure` 消息进入系统后，主流程如下：

1. 解析 `topic / dataReference / partitionCode`
2. 通过绑定关系找到 `device / monitor / ODS_DWEQ_DM_SHAFT_FLOOR_D`
3. 更新 Redis 实时状态
4. 使用 MyBatis 写入 `ODS_DWEQ_DM_RAW_DATA_D`
5. 执行规则判断
6. 命中规则时生成或合并 `alarm`
7. 按限频策略写入 `event`

### 5.2 Alarm 链路

`Alarm` 消息进入系统后，主流程如下：

1. 解析消息并提取 `iotCode / partitionId / dataReference`
2. 直接落库到 `ODS_DWEQ_DM_ALARM_RAW_D`
3. 不驱动内部 `alarm / event` 主流程（内部主流程仅由 `Measure` 触发）

## 6. 当前已实现的规则

- 定温告警 `TEMP_THRESHOLD`
- 差温告警 `TEMP_DIFFERENCE`
- 升温速率告警 `TEMP_RISE_RATE`
- 分区故障 `PARTITION_FAULT`
- 设备离线告警 `DEVICE_OFFLINE`

这些规则当前都通过 `ODS_DWEQ_DM_ALARM_RULE_D` 表做全局配置。
当前不支持按区域、监测对象、设备做差异化规则覆盖。

## 7. Redis 使用方式

Redis 当前承担三类职责：

- 最近实时状态
- 最近窗口缓存
- 活跃告警 / 告警事件限频

主要 key：

- `device:last_report:{deviceId}`
- `partition:last_measure:{partitionCode}`
- `partition:last_alarm:{partitionCode}`
- `partition:window:{partitionCode}`
- `alarm:active:{alarmType}:{partitionCode|device}`
- `alarm:last_event:{alarmType}:{partitionCode|device}`

## 8. 关键配置说明

当前几个会直接影响业务行为的配置，建议交接和上线前重点确认：

- `shaft.alarm.event-throttle-seconds`
  告警事件限频窗口，单位秒。
  同一 `monitorId + alarmType` 的持续命中，在该时间窗内不会重复写 `event`，但 `merge_count` 仍会持续增长。
  当前默认值：`300`

- `shaft.alarm.window-size`
  分区实时窗口缓存长度。
  Redis 中 `partition:window:{partitionCode}` 只保留最近 N 条摘要。

- `shaft.inspection.fixed-delay-ms`
  离线巡检周期，单位毫秒。

- `shaft.alarm.offline-threshold-seconds`
  设备离线判定阈值，单位秒。

- `spring.datasource.*`
  MySQL 连接配置。

- `spring.redis.*`
  Redis 连接配置。

### 8.0 当前配置项与实际作用

当前 `application.yml` 里的这批配置已经按下面方式生效：

- `shaft.alarm.temperature-threshold`
  定温告警默认阈值。
  当前优先读取 `ODS_DWEQ_DM_ALARM_RULE_D` 表中的 `TEMP_THRESHOLD`；若数据库未配置，则回退到此值。

- `shaft.alarm.temperature-diff-threshold`
  差温告警默认阈值。
  当前优先读取 `ODS_DWEQ_DM_ALARM_RULE_D` 表中的 `TEMP_DIFFERENCE`；若数据库未配置，则回退到此值。

- `shaft.alarm.rise-rate-threshold`
  升温速率告警默认阈值。
  当前优先读取 `ODS_DWEQ_DM_ALARM_RULE_D` 表中的 `TEMP_RISE_RATE`；若数据库未配置，则回退到此值。

- `shaft.alarm.offline-threshold-seconds`
  设备离线告警默认阈值，单位秒。
  当前优先读取 `ODS_DWEQ_DM_ALARM_RULE_D` 表中的 `DEVICE_OFFLINE`；若数据库未配置，则回退到此值。

- `shaft.alarm.window-size`
  Redis 分区窗口长度，直接生效。
  作用于 `partition:window:{partitionCode}`。

- `shaft.alarm.event-throttle-seconds`
  告警事件限频窗口，直接生效。
  作用于同一 `monitorId + alarmType` 的重复事件抽样写入。

- `shaft.cache.partition-binding-refresh-ms`
  分区绑定缓存刷新周期，直接生效。
  作用于 `dataReference / partitionCode -> device / monitor / floor` 本地缓存刷新。

- `shaft.inspection.enabled`
  是否启用离线巡检任务，直接生效。

- `shaft.inspection.fixed-delay-ms`
  离线巡检任务固定间隔，直接生效。

- `shaft.alarm.fiber-break-threshold`
  遗留配置。
  当前系统故障告警走上游 `FaultStatus -> PARTITION_FAULT`，该阈值没有进入当前执行链路。

### 8.1 事件限频说明

事件限频只作用于“持续重复命中”的事件写入，不作用于主单累计次数：

- 首次触发：一定写 `event`
- 人工处理或状态变化：一定写 `event`
- 持续重复命中：只有超过 `shaft.alarm.event-throttle-seconds` 才补写新的 `event`

因此：

- `merge_count`
  表示这条主单累计命中了多少次
- `event_count`
  表示这条主单实际写入了多少条关键事件

两者不要求相等，这是当前设计的预期行为。

## 9. MySQL 使用方式

MySQL 当前承担：

- 原始数据追溯
- 告警主单
- 告警事件
- 设备在线状态日志
- 查询接口

当前数据访问层实现方式：

- 插入、查询、更新统一使用 MyBatis Mapper
- 不再依赖 JPA Repository

`ODS_DWEQ_DM_RAW_DATA_D` 当前采用单表方案：

- 每条分区测温消息写一行 `ODS_DWEQ_DM_RAW_DATA_D`
- 不做按月拆表
- 不做自动清理任务
- 历史保留策略由外部归档/运维策略控制

### 10.7 MQTT 联调补充（双模块）

- MQTT 消息由 `iot` 模块消费后处理。
- `iot` 模块会调用 `service` 模块：
  - `POST /api/service/iot/data/batch`
  - `POST /api/service/iot/alarm/batch`
- 因此 MQTT 联调时必须同时启动 `service` 和 `iot`。

### 10.9 常见失败原因

- `Device not found`
  `iotCode` 或 `IedFullPath` 无法映射到 `device`

- `Partition binding not found`
  `dataReference / partitionCode` 与 `ODS_DWEQ_DM_MONITOR_PARTITION_BIND_D` 不一致

- HTTP `400`
  多数是入参缺字段、时间格式不对、或绑定数据未初始化

- 能上报但无告警
  先检查 `ODS_DWEQ_DM_ALARM_RULE_D` 阈值是否高于当前测试值

- `service` 已启动但 MQTT 不生效
  检查 `iot` 是否启动，且 `iot` 侧 `shaft.mq.enabled` 是否开启。
