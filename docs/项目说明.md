# 测温物联网系统项目说明

## 1. 项目现状

当前仓库已经完成一版可联调、可演示的最小可用实现，面向“分区 MQ 测温消息”模型，而不是早期的一维数组模型。

系统重点能力包括：

- 分区测温消息接收
- Redis 实时状态缓存
- `raw_data` 原始快照落库
- 规则告警生成
- 告警主单合并
- 告警事件流水
- 离线巡检任务
- 查询接口

## 2. 当前业务模型

### 2.1 消息模型

系统按两类消息处理：

- `Measure`
  分区测量摘要，包含 `MaxTemp`、`MinTemp`、`AvgTemp` 等
- `Alarm`
  分区状态摘要，包含 `AlarmStatus`、`FaultStatus`

### 2.2 绑定模型

一个设备下有多个分区，每个分区通过 `monitor_partition_bind` 绑定到：

- 监测对象 `monitor`
- 楼层 `shaft_floor`
- 分区编码 `partition_code`
- MQ 引用 `data_reference`

### 2.3 告警模型

告警采用：

- `alarm`
  告警主单
- `event`
  告警过程事件

主单和事件的语义已经拆开：

- `merge_count`
  累计触发次数
- `event_count`
  已写入的关键事件条数

## 3. 告警状态

当前告警状态码如下：

- `0`
  待确认
- `1`
  已确认
- `2`
  持续观察
- `3`
  误报
- `4`
  已关闭
- `5`
  自动恢复

系统接口返回时会同时给出 `statusCode` 和 `statusName`。

## 4. 告警合并规则

当前规则已经按业务要求实现：

- 同一监测对象、同一告警类型，在 `待确认` 状态下合并
- 不按楼层拆主单，不按分区拆主单
- `已确认 / 持续观察 / 误报 / 已关闭 / 自动恢复` 状态不再继续合并
- 后续同监测对象同类告警会新建新的待确认主单

为保证并发下不重复建待确认主单，系统使用：

- `alarm.merge_key`
- `UNIQUE KEY uk_alarm_merge_key (merge_key)`

其中：

- 待确认主单：`merge_key = monitorId:alarmType`
- 非待确认主单：`merge_key = NULL`

这保证了数据库层只允许存在一条可合并的待确认主单。

## 5. 实时数据处理链路

### 5.1 Measure 链路

`Measure` 消息进入系统后，主流程如下：

1. 解析 `topic / dataReference / partitionCode`
2. 通过绑定关系找到 `device / monitor / shaft_floor`
3. 更新 Redis 实时状态
4. 使用 JDBC 写入 `raw_data`
5. 更新 Redis 分钟聚合缓存
6. 执行规则判断
7. 命中规则时生成或合并 `alarm`
8. 按限频策略写入 `event`

### 5.2 Alarm 链路

`Alarm` 消息进入系统后，主流程如下：

1. 解析分区绑定
2. 识别 `AlarmStatus / FaultStatus`
3. 直接驱动火警 / 故障告警生成或恢复

## 6. 当前已实现的规则

- 定温告警 `TEMP_THRESHOLD`
- 差温告警 `TEMP_DIFFERENCE`
- 升温速率告警 `TEMP_RISE_RATE`
- 分区火警 `PARTITION_FIRE`
- 分区故障 `PARTITION_FAULT`
- 设备离线告警 `DEVICE_OFFLINE`

## 7. Redis 使用方式

Redis 当前承担三类职责：

- 最近实时状态
- 最近窗口缓存
- 活跃告警 / 告警事件限频

主要 key：

- `device:last_report:{deviceId}`
- `partition:last_measure:{partitionCode}`
- `partition:last_alarm:{partitionCode}`
- `partition:window:{partitionCode}`
- `alarm:active:{alarmType}:{monitorId}`
- `alarm:last_event:{alarmType}:{monitorId}`
- `minute:stat:{partitionCode}:{yyyyMMddHHmm}`

## 8. MySQL 使用方式

MySQL 当前承担：

- 原始数据追溯
- 告警主单
- 告警事件
- 分钟统计
- 设备在线状态日志
- 查询接口

高频写已经做过第一步优化：

- `raw_data` 使用 JDBC 直写
- `event` 使用 JDBC 直写
- `temp_stat_minute` 先在 Redis 聚合，再由定时任务刷库

## 9. 当前接口

### 9.1 接收接口

- `POST /api/reports/measure`
- `POST /api/reports/alarm`

### 9.2 告警接口

- `GET /api/alarms`
- `GET /api/alarms/{id}`
- `POST /api/alarms/{id}/confirm`
- `POST /api/alarms/{id}/observe`
- `POST /api/alarms/{id}/false-positive`
- `POST /api/alarms/{id}/close`

### 9.3 查询接口

- `GET /api/events`
- `GET /api/raw-data`
- `GET /api/temp-stats/minute`
- `GET /api/devices/{deviceId}/realtime`
