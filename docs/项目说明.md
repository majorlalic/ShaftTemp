# 测温物联网系统项目说明

## 1. 项目现状

当前仓库已经完成一版可联调、可演示的最小可用实现，主链路面向“分区 MQ 测温消息”模型。同时，仓库内已经预留“设备数组原始消息”链路，用于后续接入设备级一维数组原始数据。

系统重点能力包括：

- 分区测温消息接收
- 设备数组原始消息接收预案
- Redis 实时状态缓存
- `raw_data` 原始快照落库
- `device_raw_data` 设备数组原始快照落库
- 规则告警生成
- 告警主单合并
- 告警事件流水
- 离线巡检任务
- 查询接口
- 告警规则 CRUD
- 终端视图接口
- 组织树查询、模板下载、CSV 导入
- 全局规则动态生效

## 2. 当前业务模型

### 2.1 消息模型

系统当前按三类消息处理：

- `Measure`
  分区测量摘要，包含 `MaxTemp`、`MinTemp`、`AvgTemp` 等
- `Alarm`
  分区状态摘要，包含 `AlarmStatus`、`FaultStatus`
- `DeviceArray`
  设备级一维数组原始数据，包含 `values`、有效点位区间、采集时间等

### 2.2 绑定模型

一个设备下有多个分区，每个分区通过 `monitor_partition_bind` 绑定到：

- 监测对象 `monitor`
- 楼层 `shaft_floor`
- 分区编码 `partition_code`
- MQ 引用 `data_reference`

### 2.3 告警模型

告警采用：

- `alarm`
  告警主单
- `event`
  告警过程事件

主单和事件的语义已经拆开：

- `merge_count`
  累计触发次数
- `event_count`
  已写入的关键事件条数

## 3. 告警状态

当前告警状态码如下：

- `0`
  待确认
- `1`
  已确认
- `2`
  持续观察
- `3`
  误报
- `4`
  已关闭
- `5`
  自动恢复

系统接口返回时会同时给出 `statusCode` 和 `statusName`。

## 4. 告警合并规则

当前规则已经按业务要求实现：

- 同一监测对象、同一告警类型，在 `待确认` 状态下合并
- 不按楼层拆主单，不按分区拆主单
- `已确认 / 持续观察 / 误报 / 已关闭 / 自动恢复` 状态不再继续合并
- 后续同监测对象同类告警会新建新的待确认主单

为保证并发下不重复建待确认主单，系统使用：

- `alarm.merge_key`
- `UNIQUE KEY uk_alarm_merge_key (merge_key)`

其中：

- 待确认主单：`merge_key = monitorId:alarmType`
- 非待确认主单：`merge_key = NULL`

这保证了数据库层只允许存在一条可合并的待确认主单。

## 5. 实时数据处理链路

### 5.1 Measure 链路

`Measure` 消息进入系统后，主流程如下：

1. 解析 `topic / dataReference / partitionCode`
2. 通过绑定关系找到 `device / monitor / shaft_floor`
3. 更新 Redis 实时状态
4. 使用 JDBC 写入 `raw_data`
5. 更新 Redis 分钟聚合缓存
6. 执行规则判断
7. 命中规则时生成或合并 `alarm`
8. 按限频策略写入 `event`

### 5.2 Alarm 链路

`Alarm` 消息进入系统后，主流程如下：

1. 解析分区绑定
2. 识别 `AlarmStatus / FaultStatus`
3. 直接驱动火警 / 故障告警生成或恢复

### 5.3 DeviceArray 链路

`DeviceArray` 消息进入系统后，主流程如下：

1. 解析 `iotCode / IedFullPath / topic`
2. 找到 `device / monitor`
3. 计算数组 `max / min / avg`
4. 使用 JDBC 写入 `device_raw_data`

这条链路当前只做设备原始数组落库，不驱动告警，不影响现有分区 `Measure / Alarm` 主流程。

## 6. 当前已实现的规则

- 定温告警 `TEMP_THRESHOLD`
- 差温告警 `TEMP_DIFFERENCE`
- 升温速率告警 `TEMP_RISE_RATE`
- 分区火警 `PARTITION_FIRE`
- 分区故障 `PARTITION_FAULT`
- 设备离线告警 `DEVICE_OFFLINE`

这些规则当前都通过 `alarm_rule` 表做全局配置。
当前不支持按区域、监测对象、设备做差异化规则覆盖。

## 7. Redis 使用方式

Redis 当前承担三类职责：

- 最近实时状态
- 最近窗口缓存
- 活跃告警 / 告警事件限频

主要 key：

- `device:last_report:{deviceId}`
- `partition:last_measure:{partitionCode}`
- `partition:last_alarm:{partitionCode}`
- `partition:window:{partitionCode}`
- `alarm:active:{alarmType}:{monitorId}`
- `alarm:last_event:{alarmType}:{monitorId}`
- `minute:stat:{partitionCode}:{yyyyMMddHHmm}`

## 8. 关键配置说明

当前几个会直接影响业务行为的配置，建议交接和上线前重点确认：

- `shaft.alarm.event-throttle-seconds`
  告警事件限频窗口，单位秒。
  同一 `monitorId + alarmType` 的持续命中，在该时间窗内不会重复写 `event`，但 `merge_count` 仍会持续增长。
  当前默认值：`300`

- `shaft.alarm.window-size`
  分区实时窗口缓存长度。
  Redis 中 `partition:window:{partitionCode}` 只保留最近 N 条摘要。

- `shaft.inspection.fixed-delay-ms`
  离线巡检周期，单位毫秒。

- `shaft.alarm.offline-threshold-seconds`
  设备离线判定阈值，单位秒。

- `spring.datasource.*`
  MySQL 连接配置。

- `spring.redis.*`
  Redis 连接配置。

### 8.0 当前配置项与实际作用

当前 `application.yml` 里的这批配置已经按下面方式生效：

- `shaft.alarm.temperature-threshold`
  定温告警默认阈值。
  当前优先读取 `alarm_rule` 表中的 `TEMP_THRESHOLD`；若数据库未配置，则回退到此值。

- `shaft.alarm.temperature-diff-threshold`
  差温告警默认阈值。
  当前优先读取 `alarm_rule` 表中的 `TEMP_DIFFERENCE`；若数据库未配置，则回退到此值。

- `shaft.alarm.rise-rate-threshold`
  升温速率告警默认阈值。
  当前优先读取 `alarm_rule` 表中的 `TEMP_RISE_RATE`；若数据库未配置，则回退到此值。

- `shaft.alarm.offline-threshold-seconds`
  设备离线告警默认阈值，单位秒。
  当前优先读取 `alarm_rule` 表中的 `DEVICE_OFFLINE`；若数据库未配置，则回退到此值。

- `shaft.alarm.window-size`
  Redis 分区窗口长度，直接生效。
  作用于 `partition:window:{partitionCode}`。

- `shaft.alarm.event-throttle-seconds`
  告警事件限频窗口，直接生效。
  作用于同一 `monitorId + alarmType` 的重复事件抽样写入。

- `shaft.cache.partition-binding-refresh-ms`
  分区绑定缓存刷新周期，直接生效。
  作用于 `dataReference / partitionCode -> device / monitor / floor` 本地缓存刷新。

- `shaft.stat.flush-delay-ms`
  分钟统计刷库任务固定间隔，直接生效。
  作用于 Redis 分钟聚合刷入 `temp_stat_minute`。

- `shaft.inspection.enabled`
  是否启用离线巡检任务，直接生效。

- `shaft.inspection.fixed-delay-ms`
  离线巡检任务固定间隔，直接生效。

- `shaft.alarm.fiber-break-threshold`
  遗留配置。
  当前系统故障告警走上游 `FaultStatus -> PARTITION_FAULT`，该阈值没有进入当前执行链路。

### 8.1 事件限频说明

事件限频只作用于“持续重复命中”的事件写入，不作用于主单累计次数：

- 首次触发：一定写 `event`
- 人工处理或状态变化：一定写 `event`
- 持续重复命中：只有超过 `shaft.alarm.event-throttle-seconds` 才补写新的 `event`

因此：

- `merge_count`
  表示这条主单累计命中了多少次
- `event_count`
  表示这条主单实际写入了多少条关键事件

两者不要求相等，这是当前设计的预期行为。

MySQL 当前承担：

- 告警事件

### 9.1 原始数据双表现状与预案

当前主方案：

- 分区 `Measure` 消息直接写入 `raw_data`
- 这张表当前承载的是“分区摘要原始快照”

当前预备方案：

- 如果后续现场能稳定提供“设备级一维数组”消息，则使用独立表 `device_raw_data` 存储设备数组
- 不与当前分区版 `raw_data` 混存

建议原因：

- 分区摘要和设备数组是两种不同语义的数据
- 共用一张表会让字段含义和查询语义变混
- 拆表后可以分别制定保留周期、分表策略和查询入口

建议方向：

- 当前表：
  继续保留为分区摘要原始表
- 预备表：
  新增设备数组原始表，例如 `device_raw_data`

当前预备链路已经具备：

- `POST /api/reports/device-array`
- `GET /api/device-raw-data`
- `device_raw_data` 独立落库
- 不参与告警生成

查询影响：

- 当前最直接受影响的是原始数据追溯接口
- 后续如果启用双表，建议将原始数据查询明确区分为：
  - 分区摘要原始数据查询
  - 设备数组原始数据查询
- 不建议让两种来源的数据在一个接口里自动混查

当前结论：

- 现阶段继续使用分区版 `raw_data`
- 设备数组方案作为预备方案保留
- 真正确认现场能稳定提供设备数组消息后，再启用第二张原始表
