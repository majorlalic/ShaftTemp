# 测温系统交接手册（8章，含 iotservice/onlineservice）

## 第1章 系统边界与总体架构

本系统面向“竖井/楼层分区测温告警”场景，核心能力包括：

- 实时测温数据接入与处理
- Redis 实时状态缓存
- 原始数据持久化
- 告警生成、合并、状态流转
- 离线巡检告警

系统分为两个独立模块：

- `iotservice`：接入层，负责 MQTT 收包、初步解析、上送 onlineservice。
- `onlineservice`：业务层，负责规则判定、告警流程、查询接口与状态管理。

---

## 第2章 双模块设计与职责（重点）

## 2.1 iotservice

定位：协议接入与转发，不承载核心告警合并逻辑。

主要职责：

- 订阅 MQTT 主题（如 `/TMP/+/Measure`、`/TMP/+/Alarm`）。
- 识别消息类型（Measure/Alarm）。
- 解析数组或单条消息，补齐 `topic`、`iotCode`。
- 可做入站原文留痕（按当前分支实现）。
- 调用 onlineservice 接口：
  - `POST /shaft/iot/data/batch`
  - `POST /shaft/iot/alarm/batch`

交接重点：

- 主题配置、broker 连接参数、clientId 的一致性。
- topic 设备段与设备 `iot_code` 大小写严格一致。
- iotservice 与 onlineservice 的网络连通性和 baseUrl 配置。

## 2.2 onlineservice

定位：核心业务引擎。

主要职责：

- 主数据映射解析（device/monitor/floor/partition）。
- Redis 实时状态维护（最近上报、分区窗口、活跃告警）。
- 规则判断（定温、差温、升温速率、离线、分区故障）。
- 告警主单 `alarm` 的创建/合并/恢复/处警。
- 告警事件 `event` 写入与限频。
- 对外查询接口（告警、监测对象、设备、区域等）。

交接重点：

- `createOrMerge` 是主告警入口。
- `merge_count` 与 `event_count` 口径不同（累计命中 vs 事件条数）。
- 离线巡检任务是设备离线告警的来源之一。

## 第3章 数据模型与表关系（ECMS 命名）

当前表命名规则：

- 原 `ODS_DWEQ_DM_*_D` 统一改为 `ECMS_D_ST_*`

核心主数据表：

- `ECMS_D_ST_AREA`
- `ECMS_D_ST_ORG`（与 area 同步结构）
- `ECMS_D_ST_SHAFT_DEVICE`
- `ECMS_D_ST_MONITOR`
- `ECMS_D_ST_SHAFT_FLOOR`
- `ECMS_D_ST_MONITOR_DEVICE_BIND`
- `ECMS_D_ST_MONITOR_PARTITION_BIND`

运行增量表：

- `ECMS_D_ST_RAW_DATA`
- `ECMS_D_ST_ALARM_RAW`
- `ECMS_D_ST_ALARM`
- `ECMS_D_ST_EVENT`
- `ECMS_D_ST_DEVICE_ONLINE_LOG`
- `ECMS_D_ST_ALARM_RULE`

关系主线：

- `area/org -> monitor -> floor`
- `device <-> monitor`（bind）
- `monitor/device/floor <-> partition`（partition_bind）
- `alarm/event` 依赖 monitor/device/floor/partition 维度字段

## 第5章 接入链路与字段映射

MQTT 示例：

- topic：`/TMP/xinfu_16A/Measure`
- payload：包含 `PartitionId`、`dataReference`、温度字段、`timestamp`

映射优先级：

1. `iotCode + PartitionId`
2. `dataReference`
3. `partitionCode`（`dataReference` 最后一段）

映射结果：

- `device`、`monitor`、`shaft_floor`、`partition_code`

常见错误：

- `partition binding not found`

根因通常是：

- iotCode 不匹配
- partitionId/dataReference 未在绑定表配置

## 第6章 告警机制与流程

触发来源：

- 实时规则：`TEMP_THRESHOLD`、`TEMP_DIFFERENCE`、`TEMP_RISE_RATE`
- 设备规则：`DEVICE_OFFLINE`、`PARTITION_FAULT`

合并规则：

- 合并键：`monitorId + alarmType`
- 待确认状态可继续合并
- 非待确认状态新命中产生新告警

关键字段：

- `merge_count`：累计命中次数
- `event_count`：实际事件写入次数（受限频影响）

事件限频：

- 配置项：`shaft.alarm.event-throttle-seconds`

状态流转：

- `0 待确认`、`1 持续观察`、`2 待消缺`、`3 待复测`、`4 已确认`、`5 闭环`

## 第7章 核心接口分组

## 7.1 iot 接入接口

- `POST /shaft/iot/reports/measure`
- `POST /shaft/iot/reports/alarm`
- `POST /shaft/iot/data/batch`
- `POST /shaft/iot/alarm/batch`

## 7.2 业务接口

- 监测对象：`/shaft/monitor/*`
- 设备：`/shaft/device/*`
- 告警：`/shaft/alarm/*`
- 区域组织：`/shaft/areaOrg/*`

特别说明：

- `GET /shaft/monitor/list` 支持同接口切换“竖井列表/楼层列表”，并已扩展：
  - `todayAlarm`
  - `maxTemp`
  - `minTemp`
  - `avgTemp`

## 第8章 交接检查清单与排障建议

交接检查清单：

- 两模块配置文件与环境变量已移交
- 数据库表名已统一为 `ECMS_D_ST_*`
- 主数据与联调脚本可执行并有统计校验
- MQTT topic、iotCode、partition 绑定校验通过
- 告警合并与状态流转接口验证通过
- 压测脚本和人工联调流程已交接

常见排障建议：

- 先看绑定，再看规则，再看 Redis，再看告警状态。
- 告警“没合并”优先检查状态是否仍为待确认。
- 有 `merge_count` 无新增 event，多数是限频行为，不是故障。
