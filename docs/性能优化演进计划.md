# 性能优化演进计划

本文档基于当前系统的实际实现，针对“74 台设备、每台 50 个分区、每分钟每分区 1 条数据”的规模，给出性能分析和分阶段优化方案。

## 1. 规模估算

已知条件：

- 设备数：74
- 每台设备分区数：50
- 每分钟每分区 1 条 `Measure`

推导：

- 分区总数：`74 * 50 = 3700`
- 每分钟消息量：`3700`
- 每小时消息量：`222000`
- 每天消息量：`5328000`
- 平均每秒消息量：约 `61.7`

结论：

- 这个量级不是不能做
- 但不能长期沿用当前“每条消息同步完成所有数据库动作”的模式

真正的风险不是平均 TPS，而是：

- 高频同步写库
- 大表快速膨胀
- 告警重复事件风暴
- 查询直接扫大表
- 绑定关系重复查库

## 2. 当前实现的主要瓶颈

### 2.1 每条消息同步动作过多

当前一条 `Measure` 进入系统后，会同步完成：

1. 分区绑定解析
2. `raw_data` 落库
3. `temp_stat_minute` 查库再更新
4. Redis 最近状态更新
5. 规则判断
6. `alarm/event` 生成或合并
7. 设备在线状态更新

问题：

- 单条链路长
- MySQL 往返次数多
- 写放大明显

### 2.2 `raw_data` 增长过快

`raw_data` 按每次上报一行保存。

按当前规模估算：

- 日新增：约 532.8 万
- 月新增：约 1.6 亿

如果长期单表保存，会带来：

- 查询变慢
- 清理困难
- 索引维护成本高
- 数据备份和恢复时间变长

### 2.3 `temp_stat_minute` 当前是逐条查库更新

当前实现是：

- 每条消息先查这一分钟、这一分区是否已有记录
- 有则更新
- 无则插入

问题：

- 每条消息至少多一次数据库读取
- 高并发下容易放大锁竞争和写放大

### 2.4 `event` 容易膨胀

如果某分区持续异常，比如连续高温 2 小时：

- 每分钟一条消息
- 单分区单告警就可能写 120 条事件

如果多个分区持续异常，`event` 表会很快膨胀。

### 2.5 绑定关系重复查 MySQL

当前每条消息通过 `dataReference/partitionCode` 查询 `monitor_partition_bind`，再查 `device/monitor/floor`。

问题：

- 同一个分区会被反复查
- 查询结果高度重复
- 不适合作为长期高频路径

### 2.6 查询层如果直接查大表，会越来越慢

后续如果页面直接查：

- `raw_data`
- `event`

随着数据增长，性能会明显退化。

## 3. 优化目标

目标不是盲目追求架构复杂，而是保持：

- 仍然只使用 MySQL + Redis
- 仍然保持单体服务
- 实现简单、可排查、便于上线

优化后的目标是：

- Redis 扛实时状态
- MySQL 扛查询和持久化
- 高频路径减少数据库读写次数
- 历史大表按时间拆分
- 告警事件只记录有价值的状态变化

## 4. 目标数据分层

建议把数据分成三层。

### 4.1 热数据

载体：Redis

用途：

- 设备最后上报时间
- 分区最近测量摘要
- 分区最近窗口
- 活跃告警索引
- 分钟聚合缓存

特点：

- 高频读写
- 保存最近状态
- 不做长期归档

### 4.2 温数据

载体：`temp_stat_minute`

用途：

- 趋势查询
- 报表查询
- 常规页面展示

特点：

- 查询频繁
- 数据量远小于 `raw_data`
- 应作为趋势类查询的主要来源

### 4.3 冷数据

载体：`raw_data`

用途：

- 原始追溯
- 问题排查
- 事后核验

特点：

- 数据量最大
- 不应作为实时或常规查询主表

## 5. P0 必做项

P0 是上线前或上线初期必须尽快完成的优化。

### 5.1 分区绑定缓存化

目标：

- 不再每条消息都查 MySQL 绑定表

做法：

- 启动时把 `monitor_partition_bind` 载入 Redis 或本地内存缓存
- key 建议：
  - `partition:bind:{partitionCode}`
  - 或 `partition:bind:{dataReference}`

缓存内容：

- `deviceId`
- `monitorId`
- `shaftFloorId`
- `partitionCode`
- `partitionName`
- `deviceToken`
- `partitionNo`

收益：

- 主链路减少数据库查询
- 分区解析耗时稳定

### 5.2 分钟聚合前移到 Redis

目标：

- `temp_stat_minute` 不再逐条查库更新

做法：

- 每条 `Measure` 先更新 Redis 分钟聚合 key
- 例如：
  `minute:stat:{partitionCode}:{yyyyMMddHHmm}`

缓存字段：

- `deviceId`
- `monitorId`
- `shaftFloorId`
- `partitionCode`
- `maxTemp`
- `minTemp`
- `sumAvgTemp`
- `count`
- `alarmCount`

再由定时任务按分钟批量刷到 MySQL。

收益：

- 主链路减少数据库读写
- 聚合逻辑更加可控

### 5.3 `event` 限频或按状态变化写入

目标：

- 避免持续异常期间每分钟写一条 `event`

推荐策略：

- 必写事件：
  - 首次触发
  - 恢复
  - 人工确认
  - 人工关闭
  - 告警等级变化

- 可选事件：
  - 持续命中时，按 5 分钟或 10 分钟限频写入

更推荐的实现是：

- 主单 `alarm` 记录持续状态
- `event` 只记录关键状态变化

收益：

- 显著降低 `event` 表膨胀速度
- 降低无意义重复事件噪音

### 5.4 查询分层收口

目标：

- 不让业务查询直接压到 `raw_data`

原则：

- 实时页面查 Redis
- 趋势图查 `temp_stat_minute`
- 原始追溯才查 `raw_data`
- 告警列表查 `alarm`
- 告警明细查 `event`

收益：

- 减少大表扫描
- 查询性能更稳定

## 6. P1 建议项

P1 是在系统稳定后尽快做的结构优化。

### 6.1 `raw_data` 分月表

目标：

- 控制单表体量

建议：

- 按月分表，例如：
  - `raw_data_202603`
  - `raw_data_202604`

原因：

- 比日表更容易维护
- 比单表更利于归档和删除
- 更符合当前项目“简单、可排查”的要求

### 6.2 `event` 分月表

`event` 也会持续增长，建议与 `raw_data` 同样按月拆分。

### 6.3 高频写表逐步脱离 JPA

当前 JPA 适合快速实现，但不适合长期高频写。

建议优先替换：

- `raw_data`
- `event`
- `temp_stat_minute`

方式：

- `JdbcTemplate`
- 或批量 insert/update

保留 JPA 的部分：

- 配置类查询
- 管理类查询
- 中低频查询表

### 6.4 历史保留策略

建议：

- `raw_data`
  在线保留 30 天到 90 天
- `temp_stat_minute`
  在线保留 6 个月到 12 个月
- `event`
  在线保留 6 个月
- `alarm`
  长期保留

收益：

- 控制库容量
- 缩短备份和恢复时间

## 7. P2 可选项

P2 是规模继续上升时再做的优化。

### 7.1 离线巡检改为 Redis 驱动

当前 74 台设备时，全量扫问题不大。

后续如果设备数继续增加，可以改成：

- 只扫描 Redis 中的活跃设备集合
- 或维护设备最近上报时间有序集合

### 7.2 告警等级升级/降级策略

后续可以增加：

- 持续高温一定时长后升级
- 持续恢复一定时长后降级或关闭

### 7.3 更细粒度的异步写入

在不引入新中间件的前提下，可以考虑：

- 应用内批量缓冲
- 定时批量刷库

前提是：

- 可接受轻微最终一致性

## 8. 目标链路建议

### 8.1 优化前

当前主链路：

1. 查绑定
2. 写 `raw_data`
3. 查并更新 `temp_stat_minute`
4. 更新 Redis
5. 规则判断
6. 生成/合并告警
7. 写 `event`
8. 更新设备状态

### 8.2 优化后

建议主链路只做：

1. 读绑定缓存
2. 更新 Redis 最近状态
3. 轻量规则判断
4. 必要时生成/更新 `alarm`
5. 必要时写关键 `event`
6. 写 `raw_data`

异步或定时做：

1. Redis 分钟聚合刷库
2. 重复事件限频写入
3. 历史数据归档清理

## 9. 推荐优先级

如果只允许先做三件事，优先顺序如下：

1. 分区绑定缓存化
2. `temp_stat_minute` Redis 聚合
3. `event` 状态变化写入或限频写入

这三项完成后，主链路压力会先下降一大截。

## 10. 结论

当前规模下，系统不是“完全不能跑”，而是：

- 当前实现适合验证功能闭环
- 不适合长期稳定承载每天 500 万级上报

推荐方向是：

- Redis 扛实时
- MySQL 扛持久化和查询
- `raw_data/event` 分月表
- 分钟统计前移到 Redis
- 告警事件限频
- 绑定关系缓存
- 高频写逐步从 JPA 切到 JDBC

这样可以在不引入额外中间件的前提下，把当前方案提升到可持续运行的水平。
