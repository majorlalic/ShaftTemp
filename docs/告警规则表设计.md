# 告警规则表设计

## 1. 设计目标

本次规则表设计遵循以下原则：

- 只覆盖当前明确要支持的规则维护需求
- 区分“监测对象报警”和“终端报警”
- 不做过度抽象
- 支持页面做最小 CRUD
- 规则参数以“一个告警类型下配置若干数值”为核心

当前明确纳入规则表的告警类型：

- `TEMP_THRESHOLD`
- `TEMP_DIFFERENCE`
- `TEMP_RISE_RATE`
- `DEVICE_OFFLINE`
- `PARTITION_FAULT`

当前不纳入规则配置维护：

- `PARTITION_FIRE`
  原因：该告警主要由上游状态 `AlarmStatus` 直接驱动，不作为规则阈值配置项

## 2. 规则模型

规则采用单表模型，不拆主表和子表。

一条规则记录表达：

- 规则属于哪一类业务
  - 监测对象报警
  - 终端报警
- 规则针对哪个范围
  - 全局
  - 区域
  - 监测对象
  - 终端
- 规则名
- 告警类型
- 各类阈值参数
- 是否启用

## 3. 建议表名

`alarm_rule`

## 4. 建议字段

```sql
CREATE TABLE alarm_rule (
    id bigint unsigned NOT NULL COMMENT '主键ID',
    rule_name varchar(100) DEFAULT NULL COMMENT '规则名称',
    biz_type varchar(20) DEFAULT NULL COMMENT '业务类型 MONITOR/DEVICE',
    alarm_type varchar(30) DEFAULT NULL COMMENT '告警类型',
    scope_type varchar(20) DEFAULT NULL COMMENT '作用范围 GLOBAL/AREA/MONITOR/DEVICE',
    scope_id bigint unsigned DEFAULT NULL COMMENT '作用对象ID',
    level tinyint unsigned DEFAULT NULL COMMENT '告警等级',
    threshold_value decimal(10,2) DEFAULT NULL COMMENT '主阈值',
    threshold_value2 decimal(10,2) DEFAULT NULL COMMENT '副阈值',
    duration_seconds int DEFAULT NULL COMMENT '持续时长秒数',
    enabled tinyint DEFAULT 1 COMMENT '是否启用',
    remark varchar(500) DEFAULT NULL COMMENT '备注',
    deleted tinyint DEFAULT 0 COMMENT '是否删除',
    created_on datetime DEFAULT NULL COMMENT '创建时间',
    updated_on datetime DEFAULT NULL COMMENT '更新时间',
    PRIMARY KEY (id),
    KEY idx_alarm_rule_scope (biz_type, alarm_type, scope_type, scope_id, enabled, deleted)
) COMMENT='告警规则表';
```

## 5. 字段说明

### 5.1 `biz_type`

用于区分规则属于哪类报警：

- `MONITOR`
  监测对象报警
- `DEVICE`
  终端报警

建议映射关系：

- `TEMP_THRESHOLD` -> `MONITOR`
- `TEMP_DIFFERENCE` -> `MONITOR`
- `TEMP_RISE_RATE` -> `MONITOR`
- `DEVICE_OFFLINE` -> `DEVICE`
- `PARTITION_FAULT` -> `DEVICE`

### 5.2 `scope_type + scope_id`

用于表达规则作用范围。

建议首版支持：

- `GLOBAL`
  全局规则，`scope_id = null`
- `AREA`
  某区域规则
- `MONITOR`
  某监测对象规则
- `DEVICE`
  某终端规则

优先级建议后续按以下顺序匹配：

1. `MONITOR / DEVICE`
2. `AREA`
3. `GLOBAL`

首版如果想更简单，也可以先只做：

- `GLOBAL`
- `MONITOR`
- `DEVICE`

## 6. 阈值字段建议

因为当前规则本质上都是“一个告警类型对应少量数值配置”，所以不建议首版引入 JSON 配置。

建议约定：

- `threshold_value`
  主阈值
- `threshold_value2`
  副阈值
- `duration_seconds`
  持续时长

各规则使用方式：

### 6.1 `TEMP_THRESHOLD`

- `threshold_value`
  温度阈值

### 6.2 `TEMP_DIFFERENCE`

- `threshold_value`
  差温阈值

### 6.3 `TEMP_RISE_RATE`

- `threshold_value`
  升温速率阈值
- `duration_seconds`
  观察窗口秒数

### 6.4 `DEVICE_OFFLINE`

- `threshold_value`
  离线阈值秒数

### 6.5 `PARTITION_FAULT`

首版可以不依赖阈值，仅用于是否启用：

- `enabled`

如果后续需要故障持续时长过滤，可使用：

- `duration_seconds`

## 8. 建议接口

建议首版规则接口如下：

- `GET /api/alarm-rules`
  获取规则列表
- `GET /api/alarm-rules/{id}`
  获取规则详情
- `POST /api/alarm-rules`
  新增规则
- `PUT /api/alarm-rules/{id}`
  修改规则
- `POST /api/alarm-rules/{id}/enable`
  启用规则
- `POST /api/alarm-rules/{id}/disable`
  停用规则
- `DELETE /api/alarm-rules/{id}`
  逻辑删除规则

## 9. 当前建议结论

首版规则表不追求通用规则引擎，只做：

- 一个规则一条记录
- 固定阈值字段
- 固定作用域字段
- 固定业务类型字段

这样实现成本最低，也最容易维护。
