# 告警与数据链路说明

本文档基于当前代码实际实现整理，说明从 MQ 收到数据，到 Redis 更新、MySQL 持久化、告警生成与合并、离线巡检的完整流程。

## 1. 当前支持的消息类型

系统当前按分区模型处理两类消息。

### 1.1 Measure 消息

表示某个分区的一次测量快照，核心字段如下：

- `dataReference`
- `IedFullPath`
- `MaxTemp`
- `MinTemp`
- `AvgTemp`
- `MaxTempPosition`
- `MinTempPosition`
- `MaxTempChannel`
- `MinTempChannel`
- `timestamp`

示例：

```json
{
  "topic": "/TMP/shaft-dev-01_TMP_th01/Measure",
  "IedFullPath": "/IED/shaft-dev-01",
  "dataReference": "/TMP/shaft-dev-01_TMP_th01",
  "MaxTemp": 85.5,
  "MinTemp": 62.3,
  "AvgTemp": 71.8,
  "MaxTempPosition": 15.5,
  "MinTempPosition": 2.1,
  "MaxTempChannel": 1,
  "MinTempChannel": 1,
  "timestamp": "2026-03-17T09:30:00"
}
```

### 1.2 Alarm 消息

表示某个分区的上游状态快照，核心字段如下：

- `dataReference`
- `IedFullPath`
- `AlarmStatus`
- `FaultStatus`
- `timestamp`

示例：

```json
{
  "topic": "/TMP/shaft-dev-01_TMP_th01/Alarm",
  "IedFullPath": "/IED/shaft-dev-01",
  "dataReference": "/TMP/shaft-dev-01_TMP_th01",
  "AlarmStatus": true,
  "FaultStatus": false,
  "timestamp": "2026-03-17T09:31:00"
}
```

## 2. 分区映射模型

系统已经不再按“设备上一整条一维温度数组”处理，而是按“设备下多个分区”处理。

当前业务映射关系是：

- `device`
  终端设备
- `monitor`
  监控对象
- `shaft_floor`
  楼层
- `monitor_partition_bind`
  分区绑定关系

绑定表负责把 MQ 中的分区定位到业务对象：

- `dataReference`
- `partitionCode`
- `device_id`
- `monitor_id`
- `shaft_floor_id`

因此一条消息进入系统后的第一步，不是直接判规则，而是先做分区绑定解析。

## 3. 告警类型

当前实际生效的告警类型如下。

### 3.1 实时测量类告警

来源：`Measure`

- `TEMP_THRESHOLD`
  定温超限
- `TEMP_DIFFERENCE`
  分区差温异常
- `TEMP_RISE_RATE`
  升温速率异常

### 3.2 上游状态类告警

来源：`Alarm`

- `PARTITION_FIRE`
  上游告警状态告警，对应 `AlarmStatus=true`
- `PARTITION_FAULT`
  上游故障状态告警，对应 `FaultStatus=true`

### 3.3 巡检类告警

来源：定时任务

- `DEVICE_OFFLINE`
  设备离线告警

## 4. 各类告警判定流程

### 4.1 `TEMP_THRESHOLD`

输入：`Measure`

判定逻辑：

- `MaxTemp > shaft.alarm.temperature-threshold`

默认阈值见：
[application.yml](/Users/sketch/Documents/project/ShaftTemp/src/main/resources/application.yml)

### 4.2 `TEMP_DIFFERENCE`

输入：`Measure`

判定逻辑：

- `MaxTemp - MinTemp > shaft.alarm.temperature-diff-threshold`

注意：

- 这不是旧数组模型里的“点位差温”
- 这是当前分区快照里的“分区最大最小差”

### 4.3 `TEMP_RISE_RATE`

输入：`Measure`

判定逻辑：

- 读取 Redis 中该分区上一次 `last_measure`
- 计算：
  `currentMaxTemp - previousMaxTemp`
- 如果大于 `shaft.alarm.rise-rate-threshold`，则命中

### 4.4 `PARTITION_FIRE`

输入：`Alarm`

判定逻辑：

- `AlarmStatus == true`

恢复逻辑：

- `AlarmStatus == false` 时，对当前分区的 `PARTITION_FIRE` 执行恢复

### 4.5 `PARTITION_FAULT`

输入：`Alarm`

判定逻辑：

- `FaultStatus == true`

恢复逻辑：

- `FaultStatus == false` 时，对当前分区的 `PARTITION_FAULT` 执行恢复

### 4.6 `DEVICE_OFFLINE`

输入：定时巡检

判定逻辑：

- 定时任务读取 Redis `device:last_report:{deviceId}`
- 如果当前时间与最近上报时间的间隔大于 `shaft.alarm.offline-threshold-seconds`
- 则判定设备离线

恢复逻辑：

- 当前实现没有单独生成“离线恢复告警”
- 设备一旦重新收到上报，会更新设备在线状态并记录在线日志

## 5. 告警合并规则

告警模型采用：

- `alarm`
  当前告警主单
- `event`
  每次触发、恢复、人工处理的事件流水

### 5.1 合并维度

当前代码里，告警合并维度是：

- `monitor_id`
- `partition_code`
- `alarm_type`
- 状态属于 `ACTIVE` 或 `CONFIRMED`

对应代码：
[AlarmRepository.java](/Users/sketch/Documents/project/ShaftTemp/src/main/java/com/example/demo/persistence/repository/AlarmRepository.java)

### 5.2 创建新告警

如果当前分区、当前告警类型没有打开状态的告警，则：

1. 新建一条 `alarm`
2. `status = ACTIVE`
3. `merge_count = 1`
4. 写一条 `event`

### 5.3 合并已有告警

如果已存在打开状态告警，则：

1. 不新建 `alarm`
2. 更新已有 `alarm.last_alarm_time`
3. `merge_count + 1`
4. 再写一条新的 `event`
5. `event.merged_flag = 1`

### 5.4 恢复

恢复不是删除告警，而是：

1. 把 `alarm.status` 更新为 `RECOVERED`
2. 写一条 `source_type=RECOVERY` 的 `event`
3. 删除 Redis 中的活跃告警缓存

### 5.5 人工确认和关闭

当前提供两个人工动作：

- `POST /api/alarms/{id}/confirm`
- `POST /api/alarms/{id}/close`

处理规则：

- `confirm`
  把 `alarm.status` 改成 `CONFIRMED`
  并写一条 `MANUAL_CONFIRM` 事件
- `close`
  把 `alarm.status` 改成 `CLOSED`
  并写一条 `MANUAL_CLOSE` 事件
  同时清理 Redis 中该告警的活跃索引

## 6. 从 MQ 到告警和数据的完整流程

### 6.1 MQ 接入入口

RabbitMQ 消费入口：
[TemperatureReportMessageConsumer.java](/Users/sketch/Documents/project/ShaftTemp/src/main/java/com/example/demo/ingest/mq/TemperatureReportMessageConsumer.java)

当前方式：

- 监听队列 `shaft.temp.report`
- 从 routing key 或 payload 识别消息类型
- 调用 `PartitionTopicParser`
- 再进入 `ReportIngestService`

### 6.2 Measure 链路

Measure 链路如下：

1. MQ 收到消息
2. 解析 `dataReference`
3. 通过 `monitor_partition_bind` 找到：
   - `device`
   - `monitor`
   - `shaft_floor`
   - `partitionCode`
4. 构造本次分区快照
5. 从 Redis 读取该分区上一次测量摘要
6. 执行实时规则：
   - `TEMP_THRESHOLD`
   - `TEMP_DIFFERENCE`
   - `TEMP_RISE_RATE`
7. 写 `raw_data`
8. 写或更新 `temp_stat_minute`
9. 更新 Redis 最近状态和窗口
10. 更新设备在线状态
11. 对命中的规则生成或合并 `alarm`
12. 每次命中都写 `event`
13. 对本次未命中的实时规则执行恢复

### 6.3 Alarm 链路

Alarm 链路如下：

1. MQ 收到消息
2. 解析 `dataReference`
3. 通过绑定表找到业务归属
4. 把 `AlarmStatus/FaultStatus` 写入 Redis 分区状态
5. 更新设备在线状态
6. 如果 `AlarmStatus=true`
   - 生成或合并 `PARTITION_FIRE`
7. 否则
   - 恢复 `PARTITION_FIRE`
8. 如果 `FaultStatus=true`
   - 生成或合并 `PARTITION_FAULT`
9. 否则
   - 恢复 `PARTITION_FAULT`

### 6.4 离线巡检链路

离线巡检入口：
[OfflineInspectionTask.java](/Users/sketch/Documents/project/ShaftTemp/src/main/java/com/example/demo/inspection/OfflineInspectionTask.java)

流程如下：

1. 定时任务按固定间隔执行
2. 遍历 `device`
3. 优先读取 Redis 中的 `device:last_report:{deviceId}`
4. 如果 Redis 没有，则退回数据库 `device.last_report_time`
5. 超过离线阈值则认为离线
6. Redis `device:offline_level:{deviceId}` 递增
7. 防止每轮都重复打离线告警
8. 更新设备离线状态
9. 写 `device_online_log`
10. 生成或合并 `DEVICE_OFFLINE`

## 7. Redis 实时状态说明

Redis 由：
[RealtimeStateService.java](/Users/sketch/Documents/project/ShaftTemp/src/main/java/com/example/demo/realtime/RealtimeStateService.java)
和
[RedisKeyBuilder.java](/Users/sketch/Documents/project/ShaftTemp/src/main/java/com/example/demo/realtime/RedisKeyBuilder.java)
统一管理。

当前主要 key：

- `device:last_report:{deviceId}`
  设备最后上报时间

- `device:last_data:{deviceId}`
  设备最近一次测量摘要

- `partition:last_measure:{partitionCode}`
  分区最近一次测量摘要

- `partition:window:{partitionCode}`
  分区最近 N 次测量窗口

- `partition:last_alarm:{partitionCode}`
  分区最近一次状态消息

- `alarm:active:{alarmType}:{partitionCode}`
  分区维度的活跃告警索引

- `device:offline_level:{deviceId}`
  离线巡检重复触发计数

## 8. MySQL 持久化说明

### 8.1 `raw_data`

用途：

- 保存每次 `Measure` 的原始快照

当前语义：

- 不是点位数组原文
- `values_json` 存的是分区测量摘要 JSON

### 8.2 `temp_stat_minute`

用途：

- 保存按分钟聚合的分区统计

当前策略：

- 同一分钟、同设备、同监控对象、同分区合并
- `max_temp` 取更大
- `min_temp` 取更小
- `avg_temp` 做简单平均

### 8.3 `alarm`

用途：

- 保存告警主单
- 体现当前状态、首次时间、最近时间、合并次数

### 8.4 `event`

用途：

- 保存每次触发、恢复、确认、关闭的事件流水

### 8.5 `device_online_log`

用途：

- 保存设备上线、离线的状态变化日志

## 10. 当前实现边界

当前已经实现：

- RabbitMQ 队列消费入口
- Measure/Alarm 双链路
- Redis 实时状态
- `raw_data` 落库
- `temp_stat_minute` 聚合
- `alarm + event` 模型
- 告警合并、恢复、确认、关闭
- 离线巡检
- 查询接口

当前还没有实现：

- MQTT 客户端直连 broker
- 更复杂的分钟聚合统计策略
- 离线恢复事件
- 上游状态和实时测量之间的复杂联动抑制
- 绑定管理后台接口
