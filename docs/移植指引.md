# 测温链路移植指引（面向现有业务项目接入）

本文用于把当前仓库的“测温最小可用链路”移植到另一个 Java/Spring 项目中。  
当前链路口径：`MQTT Measure -> Redis实时状态 -> raw_data月表 -> 规则告警 -> alarm/event`，`Alarm` 主题仅入 `alarm_raw`。

## 1. 移植范围

建议移植的能力：

- 设备级 topic + 分区数组消息消费
- 分区绑定解析（`device + partitionId` 优先）
- 原始数据月表写入（`raw_data_yyyyMM`）
- 告警规则计算 + 告警合并（`merge_key` 唯一）
- 事件限频写入（`event-throttle-seconds`）
- 离线巡检（`DEVICE_OFFLINE`）

已下线，不建议迁移：

- `temp_stat_minute` 分钟统计整套逻辑（已从本项目移除）

## 2. 前置条件

运行环境：

- JDK 8+

### 6.3 分区到楼层映射

通过 `monitor_partition_bind` 完成：

- 优先：`device_id + partition_id`
- 回退：`data_reference` / `partition_code`

结论：同一设备多分区时，每个分区可稳定映射到不同 `shaft_floor`。

## 7. 关键数据流

Measure 主链路：

1. MQTT 收到 `Measure`
2. 解析 `iotCode + partitionId + dataReference`
3. 绑定解析到 `device/monitor/floor`
4. 更新 Redis：最近上报、最近摘要、窗口
5. 写 `raw_data_yyyyMM`
6. 规则判断（温度阈值、差温、升温速率）
7. `alarm` 新建或合并（待确认态）
8. `event` 按限频写入

Alarm 主题链路：

1. MQTT 收到 `Alarm`
2. 原样入 `alarm_raw`
3. 不触发内部 `alarm/event` 流程

离线巡检链路：

1. 定时扫描 `device:last_report:{deviceId}`
2. 超阈值触发 `DEVICE_OFFLINE`
3. 写 `alarm/event` 并合并到待确认主单

## 8. 迁移落地步骤（推荐）

1. 先导入 SQL，确认绑定/规则基础数据齐全。
2. 将 `ingest + alarm + realtime + inspection + persistence` 模块接入目标项目。
3. 先关闭 MQTT，仅用 HTTP 调试入口跑通链路。
4. 开启 MQTT，接入真实 broker/topic。
5. 开启巡检任务，验证离线告警。
6. 压测并调优线程池、连接池、Redis。

## 9. 联调验收清单

功能验收：

- 发 1 条高温 `Measure`，`raw_data_yyyyMM` 有记录
- 命中规则后 `alarm` 新建，`event` 生成
- 同类型重复上报，`merge_count` 增长，主单不重复
- `Alarm` 主题上报后，仅 `alarm_raw` 增长
- 停止上报后，离线巡检生成 `DEVICE_OFFLINE`

数据验收：

- `alarm.merge_key` 只有待确认主单有值
- `event_count <= merge_count`
- `alarm.content` 与 `event.content` 都有可读告警文案

## 10. 性能与容量建议

- `raw_data` 保持月表策略，避免单表膨胀
- 高频写入维持 JDBC（不要回退到 JPA）
- `event-throttle-seconds` 先用 300 秒，按现场噪声调
- Redis 只做实时态，不做大体量历史存储

## 11. 常见迁移坑

- 只迁了代码没迁唯一索引：会出现待确认主单重复
- 绑定表缺 `partition_id`：多分区映射不稳定
- `topic` 解析与现场命名不一致：`iotCode` 解析失败
- 忘记 `raw_data_template`：月表自动创建失败
- 规则表没初始化：测温上报无告警

## 12. 回滚策略

建议灰度开关：

- `shaft.mq.enabled=false`：快速关闭实时消费
- `shaft.inspection.enabled=false`：快速关闭离线巡检

保底策略：

- 保留 HTTP 调试入口用于故障隔离
- 关键写链路（`raw_data`、`alarm`、`event`）优先保证可观测日志
